# 커피 주문 앱

## 1. 프로젝트 개요

### 1.1 프로젝트명
커피 주문 앱

### 1.2 프로젝트 목적
사용자가 커피 메뉴를 주문하고, 관리자가 주문을 관리할 수 있는 간단한 풀스텍 웹 앱

### 1.3 개발 범위
- 주문하기 화면(메뉴 선택 및 장바구니 기능)
- 관리자 화면(재고 관리 및 주문 상태 관리)
- 데이터를 생성/조회/수정/삭제할 수 있는 기능

## 2. 기술 스택
- 프런트엔드: HTML, CSS, 리액트, 자바스크립트
- 백엔드: Node.js, Express
- 데이터베이스: PostgreSQL

## 3. 기본 사항
- 프런트엔드와 백엔드를 따로 개발
- 기본적인 웹 기술만 사용
- 학습 목적이므로 사용자 인증이나 결제 기능은 제외
- 메뉴는 커피 메뉴만 있음

## 4. 주문하기 화면 PRD (프런트엔드 UI)

### 4.1 목적 및 사용자 목표
- **목적:** 고객이 커피 메뉴를 확인하고, 옵션을 선택한 뒤 장바구니에 담아 주문을 확정할 수 있는 단일 화면을 제공한다.
- **사용자 목표:** 메뉴·가격·옵션을 한눈에 보고, 담기 → 장바구니 확인 → 주문하기까지 끊김 없이 완료한다.

### 4.2 화면 구성 (와이어프레임 대응)
화면은 세 영역으로 구분한다.

| 영역 | 역할 |
|------|------|
| 헤더 | 브랜드 식별 및 다른 화면(관리자)으로의 이동 |
| 메뉴 영역 | 상품 카드 그리드(가로 배치), 옵션 선택 및 담기 |
| 장바구니 영역 | 담은 상품 목록, 수량·금액, 총액, 주문 확정 버튼 |

### 4.3 헤더
- **브랜드:** 좌측 상단에 앱명 **「COZY」**를 표시한다.
- **탭/링크:** 우측에 **「주문하기」**, **「관리자」** 두 항목을 둔다.
  - 현재 화면이 주문하기일 때 **「주문하기」**는 시각적으로 활성 상태임을 나타낸다(예: 테두리 또는 강조 스타일).
  - **「관리자」**는 비활성 링크 스타일로 표시하되, 클릭 시 관리자 화면으로 라우팅한다.
  - 관리자 화면에 있을 때는 **「관리자」**가 활성 스타일이고 **「주문하기」**는 일반 링크로 동작한다(5.3 참고).

### 4.4 메뉴 영역 (상품 카드)
- **레이아웃:** 사용 가능한 메뉴를 **카드 형태**로 나열한다. 와이어프레임 기준으로는 가로로 여러 카드가 보이는 형태를 목표로 한다(반응형에서는 줄바꿈 허용).
- **카드당 포함 요소**
  - **이미지:** 메뉴 대표 이미지 영역(이미지 없을 때는 플레이스홀더).
  - **상품명:** 예) 아메리카노(ICE), 아메리카노(HOT), 카페라떼.
  - **가격:** 기본 가격을 원화 형식으로 표시한다(예: 4,000원).
  - **설명:** 한 줄 또는 짧은 문단의 **간단 설명** 텍스트.
  - **옵션:** 체크박스로 다중 선택 가능.
    - 각 옵션에는 **라벨**과 **추가 금액**을 표시한다(예: 샷 추가 (+500원), 시럽 추가 (+0원)).
    - 옵션 목록과 가격은 백엔드/설정 데이터에 따라 동적으로 렌더링할 수 있도록 설계한다.
  - **행동 버튼:** **「담기」** — 현재 카드에서 선택된 옵션 조합으로 장바구니에 한 줄 추가한다.

### 4.5 장바구니 영역
- **제목:** **「장바구니」**.
- **라인 아이템:** 각 항목에 다음을 표시한다.
  - **상품명 + 선택된 옵션 요약:** 예) `아메리카노(ICE) (샷 추가) X 1`, `아메리카노(HOT) X 2`.
  - **해당 라인 금액:** 옵션 포함 계산된 소계(예: 4,500원, 8,000원).
- **수량:** 와이어프레임상 `X 2` 형태로 표현되므로, 동일 메뉴·동일 옵션 조합이 반복 담길 때는 **한 줄로 합산**하고 수량을 표시하는 동작을 PRD 기본값으로 한다(합산 규칙은 4.8 참고).
- **총액:** **「총 금액」** 라벨과 함께 장바구니 전체 합계를 우측 정렬 등으로 강조한다.
- **주문 확정:** **「주문하기」** 버튼 — 탭의 「주문하기」와 구분되도록 **장바구니 하단의 주요 CTA**로 배치한다. 클릭 시 주문 생성 플로우(백엔드 API 호출 등)로 이어진다.

### 4.6 사용자 플로우
1. 사용자가 메뉴 카드에서 옵션을 체크한다.
2. **「담기」**를 누르면 해당 구성이 장바구니에 반영되고, 장바구니 목록·총액이 갱신된다.
3. 필요 시 다른 메뉴를 같은 방식으로 추가한다.
4. 장바구니에서 총액을 확인한 뒤 **「주문하기」**(하단 버튼)를 눌러 주문을 완료한다.

### 4.7 상호작용 및 UI 상태
- 옵션 미선택도 **기본 구성(옵션 없음)**으로 담을 수 있어야 한다(체크박스는 모두 해제 가능).
- **「담기」** 직후 피드백: 최소한 장바구니 갱신으로 상태가 바뀌는 것이 보이도록 한다(선택: 토스트/짧은 메시지는 구현 선택 사항).
- 장바구니가 비어 있을 때 **「주문하기」**는 비활성화하거나, 클릭 시 안내 메시지를 띄우는 등 **빈 주문 방지** 동작을 정의한다.
- 관리자 화면과의 전환 시 **주문하기 화면의 장바구니 상태**는 학습 범위에 따라 세션/로컬 유지 여부를 결정하되, PRD상으로는 “같은 브라우저 세션에서 유지”를 권장한다.

### 4.8 가격·합산 규칙
- **라인 금액** = (상품 기본가 + 선택된 모든 옵션의 추가가 합) × 수량.
- **총 금액** = 모든 라인 금액의 합.
- **동일 상품·동일 옵션 집합**으로 **「담기」**를 여러 번 누르면 기존 라인의 **수량을 증가**시키고 소계를 다시 계산한다.
- 금액 표시는 **천 단위 구분**과 **「원」** 접미사를 사용한다.

### 4.9 범위 밖 (프로젝트 3장과 정합)
- 실제 결제 PG 연동, 로그인/회원, 배송지 입력은 포함하지 않는다.
- 주문 완료 후 화면(성공 모달, 주문 번호 안내 등)은 별도 PRD 또는 동일 문서의 후속 섹션에서 정의할 수 있다.

### 4.10 수용 기준 (요약)
- 헤더에 COZY, 주문하기(활성), 관리자 링크가 요구사항대로 보인다.
- 메뉴 카드에 이미지·이름·가격·설명·옵션 체크박스·담기가 있다.
- 담기 후 장바구니에 옵션이 반영된 라벨과 금액·총액이 정확히 갱신된다.
- 빈 장바구니에서 주문 확정이 불가하거나 사용자에게 안내된다.
- 총액 계산이 4.8의 규칙과 일치한다.

## 5. 관리자 화면 PRD (프런트엔드 UI)

### 5.1 목적 및 사용자 목표
- **목적:** 매장 운영자가 한 화면에서 **주문 흐름 요약**, **메뉴별 재고**, **들어온 주문 처리**를 수행할 수 있게 한다.
- **사용자 목표:** 대시보드 숫자로 현황을 파악하고, 재고를 조정하며, 신규 주문에 대해 접수 액션까지 끊김 없이 처리한다.

### 5.2 화면 구성 (와이어프레임 대응)
세로 **단일 컬럼** 레이아웃으로, 상단 헤더 아래에 **세 개의 구역**을 둔다. 각 구역은 **연한 테두리·약한 모서리 라운드**의 박스로 구분한다(와이어의 미니멀 스타일).

| 영역 | 역할 |
|------|------|
| 헤더 | 브랜드 및 주문하기/관리자 전환 |
| 관리자 대시보드 | 주문 상태별 집계 한 줄 요약 |
| 재고 현황 | 메뉴별 수량 표시 및 수동 증감 |
| 주문 현황 | 주문 목록·상세·접수 액션 |

### 5.3 헤더
- **브랜드:** 좌측 상단 **「COZY」**.
- **탭/링크:** 우측 **「주문하기」**, **「관리자」**.
  - 관리자 화면일 때 **「관리자」**에 활성 표시(예: 테두리), **「주문하기」**는 클릭 시 주문 화면으로 이동한다.

### 5.4 관리자 대시보드
- **제목:** **「관리자 대시보드」**.
- **내용:** 한 줄 텍스트로 **실시간에 가깝게** 갱신되는 집계를 표시한다. 항목 사이는 와이어처럼 **`/`** 등 구분자로 나열한다.
  - **총 주문:** 시스템에 누적된 주문 건수(또는 정책에 따라 ‘전체’ 정의—MVP에서는 **생성된 주문 전체 수**로 정의 가능).
  - **주문 접수:** 상태가 **「주문 접수」**인 주문 건수.
  - **제조 중:** 상태가 **「제조 중」**인 주문 건수.
  - **제조 완료:** 상태가 **「제조 완료」**인 주문 건수.
- 집계 정의는 백엔드와 동일 스키마를 기준으로 하며, 화면은 API(또는 동등한 소스) 결과를 그대로 반영한다.
- **상태 모델(권장):** 한 주문은 한 시점에 아래 중 하나에만 속한다.  
  `신규(또는 접수 대기)` → **주문 접수** → **제조 중** → **제조 완료**  
  와이어에는 목록 행에 **「주문 접수」** 버튼만 있으므로, MVP에서는 **신규(접수 대기)** 주문에 대해 이 버튼을 노출하고, 클릭 시 **「주문 접수」** 상태로 전환한다. **제조 중·제조 완료**로 넘기는 UI는 와이어에 없으나, 동일 목록 행에서 상태에 따라 버튼 라벨을 바꾸는 방식(예: "제조 시작", "제조 완료")으로 확장하는 것을 허용한다.

### 5.5 재고 현황
- **제목:** **「재고 현황」**.
- **레이아웃:** 메뉴당 **가로 카드** 하나(와이어: 세 메뉴 예시).
- **카드 내용**
  - **상품명:** 주문하기 화면과 동일한 메뉴 식별자(예: 아메리카노(ICE), 아메리카노(HOT), 카페라떼).
  - **수량:** **「N개」** 형식(예: 10개).
  - **조작:** **`+`**, **`-`** 버튼으로 재고를 **정수 단위**로 수동 조정한다.
- **`-` 동작:** 0 미만으로 내려가지 않게 한다(하한 0 클립 또는 버튼 비활성).
- **`+` 동작:** 상한이 정책상 필요하면(예: 999) 제한할 수 있으나, 학습 앱에서는 생략 가능.
- 조작 즉시 UI가 갱신되고, 서버 저장 실패 시 사용자에게 알리고 이전 값으로 되돌린다(선택: 낙관적 업데이트).

### 5.6 주문 현황
- **제목:** **「주문 현황」**.
- **행(주문 단위):** 각 주문을 한 줄(또는 한 카드)로 표시한다.
  - **일시:** 주문 생성 시각(예: 7월 31일 13:00). 로케일·포맷은 일관되게 적용한다.
  - **주문 내용:** 품목 요약(예: `아메리카노(ICE) x 1`). 복수 라인이면 줄바꿈 또는 콤마 등으로 합쳐 표기 규칙을 정한다.
  - **금액:** 해당 주문 합계(예: 4,000원). 옵션이 있는 주문은 4장 주문하기 PRD의 가격 규칙과 동일하게 합산된 금액을 표시한다.
  - **액션:** 행 우측에 **「주문 접수」** 버튼. **신규(접수 대기)** 등에서만 노출하는 것을 기본으로 하며(5.4), 클릭 후 목록·대시보드 집계가 갱신된다.

### 5.7 사용자 플로우
1. 관리자가 화면을 열면 대시보드 숫자·재고·주문 목록을 본다.
2. 재고가 부족하면 해당 카드에서 **`+`** / **`-`**로 조정한다.
3. 신규 주문 행에서 **「주문 접수」**를 눌러 상태를 진행시킨다(및 필요 시 이후 상태 버튼으로 확장).

### 5.8 주문하기 화면과의 연동 (요약)
- 주문하기에서 확정된 주문이 **주문 현황** 목록과 **총 주문** 등 집계에 반영된다.
- **재고**는 주문 확정 시 차감할지, 제조 완료 시 차감할지 등은 비즈니스 규칙으로 정하되, PRD 기본 제안은 **주문 확정 시 재고 차감**이다(관리자 수동 `+`/`-`와 함께 일관되게 설계).

### 5.9 범위 밖 (프로젝트 3장과 정합)
- 관리자 **로그인·권한·역할**은 포함하지 않는다(동일 URL로 접근 가능한 학습용 화면).
- 정산·영수증·알림(푸시·SMS)은 포함하지 않는다.

### 5.10 수용 기준 (요약)
- 헤더에 COZY, 주문하기, 관리자(활성)가 요구사항대로 보인다.
- 대시보드에 총 주문·주문 접수·제조 중·제조 완료가 한 줄로 표시되고 데이터와 일치한다.
- 재고 카드에 메뉴명·개수·`+`/`-`가 있으며 0 미만으로 내려가지 않는다.
- 주문 행에 일시·내용·금액·(조건 충족 시)「주문 접수」가 있고, 액션 후 UI가 갱신된다.

## 6. 백엔드 PRD

### 6.1 목적 및 범위
- **목적:** 주문하기·관리자 UI가 공유할 **메뉴·옵션·재고·주문 데이터**를 PostgreSQL에 일관되게 저장하고, Node.js·Express REST API로 조회·갱신한다.
- **범위:** 본 장은 데이터 모델, 스키마를 전제로 한 **사용자(시스템) 흐름**, 그리고 **필수 API**를 정의한다. 구현 세부(URL 접두사, 인증 헤더 등)는 학습 목적에 맞게 최소화해도 된다.
- **정합:** 기술 스택은 §2와 같으며, 인증·결제는 §3 및 §5.9와 같이 **제외**한다.

### 6.2 데이터 모델

#### 6.2.1 Menus (메뉴 / 커피)
메뉴 한 건이 하나의 행(record)으로 저장된다.

| 필드 | 설명 |
|------|------|
| 식별자 | 고유 ID (PK) |
| 이름 | 커피 이름(예: 아메리카노(ICE)) |
| 설명 | 메뉴 설명 텍스트 |
| 가격 | 기본 단가(정수, 원 단위 권장) |
| 이미지 | 이미지 URL 또는 저장 경로(문자열) |
| 재고 수량 | 현재 재고(정수, 0 이상). 주문 처리·관리자 조정 시 갱신 |

> **표시 규칙:** 주문하기 화면에는 이름·설명·가격·이미지·연결된 옵션을 사용한다. **재고 수량은 고객 화면에는 노출하지 않고**(§4 요구와 정합), **관리자 화면의 재고 현황**에서만 노출한다.

#### 6.2.2 Options (옵션)
옵션은 **특정 메뉴에 속한다**(메뉴와 1:N 관계).

| 필드 | 설명 |
|------|------|
| 식별자 | 고유 ID (PK) |
| 이름 | 옵션 이름(예: 샷 추가, 시럽 추가) |
| 옵션 가격 | 해당 옵션 선택 시 추가되는 금액(정수, 원 단위; 0원 가능) |
| 연결 메뉴 | 소속 메뉴 ID (Menus FK) |

복수 선택이 허용되면 클라이언트는 선택한 옵션 ID 목록을 주문 줄에 포함한다.

#### 6.2.3 Orders (주문)
주문 헤더와 **줄 단위 내용**(메뉴·수량·옵션·금액)을 저장한다.

**주문 헤더(예: `orders` 테이블)**

| 필드 | 설명 |
|------|------|
| 식별자 | 고유 주문 ID (PK) |
| 주문 일시 | 생성 시각(타임스탬프, 서버 기준 권장) |
| 상태 | `주문 접수` \| `제조 중` \| `완료` (또는 `제조 완료` 등 프런트와 동일 레이블로 매핑) |
| 주문 합계 금액 | 모든 줄 금액의 합(정수 원) |

**주문 줄(예: `order_items` 또는 동등)**

한 주문은 여러 줄을 가진다. 각 줄에 다음을 둔다.

| 필드 | 설명 |
|------|------|
| 소속 주문 | 주문 ID (FK) |
| 메뉴 | 메뉴 ID (FK); 표시 안정화를 위해 **주문 시점 스냅샷**(메뉴 이름·기본 가격 등) 저장을 허용 |
| 수량 | 정수 ≥ 1 |
| 옵션 | 선택된 옵션 ID 목록(정규화 시 별도 `order_item_options` 테이블, 또는 JSON 등 구현 선택) |
| 금액 | 해당 줄 **소계**(기본가+선택 옵션가)×수량 규칙은 §4.8과 동일해야 함 |

> **상태 초기값:** 새로 저장된 주문의 상태는 **「주문 접수」**이다. 관리자가 상태를 진행시키면 **「제조 중」→「완료」**(프런트의 「제조 완료」와 동일 의미로 통일 가능) 순으로 변경한다.

### 6.3 데이터 스키마를 전제로 한 사용자·시스템 흐름

1. **메뉴 표시:** 서버는 Menus 및 연결된 Options를 조합해 내려준다. 프런트는 이를 받아 브라우저에 카드 형태로 표시한다. **재고 수량**은 같은 Menus 레코드 기준이지만, **고객용 응답에서는 재고를 생략**하거나 내려주더라도 UI에서 숨긴다. **관리자용 메뉴·재고 API**에서는 재고를 포함해 재고 현황을 그린다.
2. **장바구니:** 사용자가 커피·옵션·수량을 선택하는 동안은 브라우저 로컬 상태로 유지하고(§4와 동일), 서버에는 아직 저장하지 않는다.
3. **주문 확정:** 사용자가 장바구니에서「주문하기」를 클릭하면, 클라이언트는 줄 목록·수량·선택 옵션을 담아 **주문 생성 API**를 호출한다. 서버는 **재고 검증**(메뉴 단위 필요 수량 ≤ 현재 재고) 후, 하나의 트랜잭션에서 **Orders(및 줄) 저장**과 **Menus 재고 차감**을 수행한다. 실패 시 롤백하고 오류를 반환한다.
4. **관리자 주문 현황:** 관리자 화면은 Orders 목록을 조회해 「주문 현황」에 일시·주문 내용(메뉴·수량·옵션·금액 요약)·합계를 표시한다. 기본 상태는 **주문 접수**이며, 관리자가 상태 전환을 요청하면 **주문 접수 → 제조 중 → 완료**로 업데이트한다(버튼 라벨은 프런트와 맞춤).

### 6.4 API 설계

다음은 REST 관점의 **필수 엔드포인트**이다. 실제 경로·이름은 구현 일관성만 유지하면 된다.

| 목적 | 메서드·경로(예시) | 요약 |
|------|------------------|------|
| 커피 메뉴 목록 조회 | `GET /api/menus` | DB의 Menus(및 필요 시 옵션)를 불러와 주문하기 화면에 맞게 반환한다. 고객 앱이라면 재고 필드 생략 권장. |
| 관리자용 메뉴·재고 | `GET /api/admin/menus` 등(선택) | 재고 수량 포함. 또는 동일 목록에 `?includeStock=true` 등으로 분기 가능. |
| 주문 생성 | `POST /api/orders` | 요청 본문에 줄 배열: 메뉴 ID, 수량, 선택 옵션 ID 목록 등. 서버가 줄 금액·합계를 검증·저장하고, **해당 메뉴 재고를 차감**한다. |
| 주문 단건 조회 | `GET /api/orders/:id` | 주문 ID를 받아 헤더·줄 상세·상태를 반환한다. |
| 주문 상태 변경 | `PATCH /api/orders/:id/status` 등(관리자) | 「주문 접수」→「제조 중」→「완료」 진행 시 사용. |
| 재고 수동 조정 | `PATCH /api/menus/:id/stock` 등(관리자) | 관리자 `+`/`−` 반영 시 Menus 재고 갱신(0 미만 불가). |

**요구사항과의 직접 대응**
- 사용자가 「주문하기」화면으로 들어오면(또는 진입 시) **데이터베이스에서 커피 메뉴 목록을 불러와** 표시한다 → **`GET` 메뉴 목록**으로 충족.
- 사용자가 「주문하기」버튼으로 확정하면 **주문을 DB에 저장**하고 → **`POST /api/orders`**.
- 저장 시점에 **주문 수량만큼 메뉴별 재고를 수정**한다 → 주문 생성 트랜잭션 내 Menus 업데이트.
- 주문 ID로 **해당 주문 정보를 조회**한다 → **`GET /api/orders/:id`**.

### 6.5 오류·검증 (요약)
- 주문 생성: 재고 부족 시 4xx와 명확한 메시지(프런트 토스트와 연동).
- 재고 숫자: 음수 저장 금지.
- 상태 전환: 허용되지 않은 전환(예: 완료에서 주문 접수로 되돌리기)은 거부할지 정책으로 명시하면 된다(학습 앱에서는 엄격할 필요 없음).

### 6.6 수용 기준 (백엔드·연동 요약)
- 메뉴 API로 주문하기 UI를 채울 수 있다.
- 주문 생성 API 동작 후 DB에 Orders·줄이 남고, Menus 재고가 감소한다.
- 주문 단건 조회가 가능하다.
- 관리자는 주문 목록·상태 갱신·재고 수정을 위한 최소 API로 화면 요구를 충족할 수 있다.
